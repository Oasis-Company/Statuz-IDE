# Critical Review ÔÇö Oasis Team's Post-Void Takeover Changes (Statuz IDE)

**Scope:** Changes made by the Oasis Company team after forking Void Editor and rebranding it to Statuz IDE (working tree + recent `oasis` commits, through `4dc73934`).
**Verdict:** The rebrand is *cosmetic at the service layer and largely absent at the user-facing layer*. As it stands, Statuz IDE would still present itself to users as "Void" in notifications, settings, tooltips, and the update flow, and its update path would pull a third party's binaries. **Not safe to ship.** The engine integration, by contrast, is a clean and well-structured stub.

---

## 0. How the build actually works (this explains most of the failures)

- `src/vs/workbench/contrib/statuz/browser/react/tsup.config.js` compiles **`./src2/**`** (not `./src/**`) into `react/out/`.
- `react/build.js` *generates* `src2/` from `src/` via `npx scope-tailwind ./src -o src2/ -s statuz-scope -c styles.css -p "statuz-"`.
- `scope-tailwind` only rewrites **CSS class scopes** (`void-scope` Ôćĺ `statuz-scope`, utility prefixes). It does **not** rename component files, import paths, or arbitrary string/data attributes such as `data-tooltip-id="void-tooltip"`.
- Both `src2/` and `out/` are committed to git.

**Consequence:** renaming `VoidX.tsx` Ôćĺ `StatuzX.tsx` in `src/` does **not** affect what ships, because the build consumes `src2/`, which is regenerated by a tool that ignores filenames and those string literals. The rebrand of component names in `src/` is effectively vapor ÔÇö it never reaches the running app.

## 1. CRITICAL ÔÇö ship-blockers

### C1. Auto-update fetches Void's binaries
`src/vs/workbench/contrib/statuz/electron-main/statuzUpdateMainService.ts:98`
```ts
const response = await fetch('https://api.github.com/repos/voideditor/binaries/releases/latest');
```
- On update check the app tells the user *"A new version of Void is available!"* (lines 115, 130) and *"Void is up-to-date!"* (119). The result type is even still named `VoidCheckUpdateRespose` (typo: "Respose").
- If auto-update were ever enabled, it would download/install **Void's** binaries, not Statuz's ÔÇö potentially overwriting the product with an unrelated upstream or failing outright. The Oasis-Company repo is never consulted.
- **Fix:** point at the Oasis-Company binaries repo; rebrand all messages; rename the type.

### C2. The running UI is still "Void"
The shipped bundle (`src2/`, `out/`) and even `src/` still say Void:
- `react/src/markdown/ChatMarkdownRender.tsx:100` Ôćĺ `'data-tooltip-id': 'void-tooltip'`
- `react/out/**` Ôćĺ `void-tooltip`, `void-scope`, `void-chats.json`, `void-settings.json`, `void-max-w-[300px]`
- `common/metricsService.ts:62,71` Ôćĺ "Void: Log Debug Info", "Void Debug info:"
- `electron-main/llmMessage/sendLLMMessage.ts:81` Ôćĺ "...wrong endpoint in Void's Settings..."
- `electron-main/llmMessage/sendLLMMessage.impl.ts:171,375,820` Ôćĺ "Void providerName was invalid", "Void: Response from model was empty."
- `browser/fileService.ts:20` Ôćĺ "Void: Copy Prompt"
- `browser/editCodeService.ts` Ôćĺ "Void Agent", `VoidFileSnapshot`, errors "Void 1", "Void: diff.type not recognized", "Void: We ran Fast Apply..."
- `electron-main/mcpChannel.ts:104`, `sendLLMMessageChannel.ts:84` Ôćĺ "Void sendLLM: command not recognized."
- `electron-main/statuzUpdateMainService.ts:82` Ôćĺ "Restart Void to update!"
- Types still `Void*`: `VoidFileSnapshot`, `VoidDirectoryItem`, `VoidStaticModelInfo`, `VoidStaticProviderInfo`, `VoidCheckUpdateRespose`.

**Impact:** a user running Statuz IDE sees "Void" everywhere. The rebrand's primary goal is unmet.

## 2. MAJOR

### M1. IPC channel IDs remain `void-channel-*`
`app.ts` registers `void-channel-metrics`, `void-channel-update`, `void-channel-llmMessage`, `void-channel-scm`; the browser side consumes the same. The service *classes* were renamed `Statuz*`, but the channel *namespace* was not. Internally consistent today (so it runs), but:
- Contradicts the roadmap's "internal references voidÔćĺstatuz: In Progress" ÔÇö it is not done.
- Latent desync hazard on any future channel rename or upstream merge.
- The variable `voidUpdatesChannel` was renamed in declaration, but the string id stayed `'void-channel-update'`.

### M2. `.voidrules` rename incomplete
Roadmap claims `.voidrules` Ôćĺ `.statuzrules` "Done", but `browser/convertToLLMMessageService.ts:549` still defines `_getVoidRulesFileContents()` and uses `voidRulesFileContent`. Rules files won't load under the new name.

### M3. `product.ts` still carries `voidVersion?: string`
`src/vs/base/common/product.ts:59` ÔÇö leftover Void-named product field.

### M4. Generated `src2/` and `out/` are committed
They are build artifacts that drift from `src/` and produce enormous, noisy diffs. They should be gitignored; only `src/` should be source of truth. Committing them also means the stale Void-branded `out/` ships until rebuilt.

### M5. Engine stub will not compile when "uncommented" for Phase 1
`common/engine/statuzEngineService.ts`: the commented `_callEngine` references `this._mainProcessService`, but the constructor only injects `ILogService`. Uncommenting as-is Ôćĺ TS error (property does not exist). The engine channel is also registered (`app.ts:1259`) but currently unused (the stub throws before contacting IPC) ÔÇö dead registration until Phase 1.

## 3. MINOR / HYGIENE

- **`as any` casts** erode type safety: `statuzSettingsService.ts:314`, `chatThreadService.ts:678,712` (dynamic `[toolName](... as any)` dispatch), `Settings.tsx:1107,1110` `dangerousSetState(json as any)`. `dangerousSetState` is a foot-gun.
- **`console.log` in main process:** `metricsMainService.ts:128,138` ("User is opted out of basic Void metrics?", "Void posthog metrics info:") ÔÇö dev logging left in a production path, and Void-branded.
- **Wrong support link:** `editCodeService.ts:293` directs users to `github.com/voideditor/void/issues/new`.
- **Typo:** `VoidCheckUpdateRespose` (Respose).
- **Roadmap overclaims:** "No VS Code core modifications beyond brand" and "Upstream merges conflict-free outside contrib/statuz/" ÔÇö `app.ts` is modified (inherited from Void, but the constraint is nonetheless violated by the fork). This should be acknowledged rather than asserted.

## 4. WHAT'S ACTUALLY GOOD

- Engine service interface (`IStatuzEngineService`) is well-designed, matches the roadmap API surface, cleanly separated, with clear docstrings and a coherent stub story.
- Engine IPC channel is correctly registered in `app.ts` (`statuz-channel-engine`).
- Service-identifier renames in `app.ts` are consistent (imports + `services.set` + channel wiring).
- Commit messages follow conventional-commit style.
- `product.json` brand fields are clean (no void leakage).

## 5. RECOMMENDED GATING BEFORE ANY RELEASE

1. **Block release** until C1 (update endpoint) and C2 (user-facing Void strings) are fixed.
2. **Decide the canonical build source:** keep `src/` as the only hand-written source; gitignore `src2/` and `out/`; ensure the scope-tailwind step also rewrites `void-tooltip` Ôćĺ `statuz-tooltip`, and that component renames propagate (or stop relying on `src2` mirroring).
3. **Add a CI guard:** fail the build if `void` / `Void` / `voideditor` appears outside an explicit allowlist.
4. **Fix M1ÔÇôM5.**
